1. Introducción
La etapa de preparación preuniversitaria genera un volumen masivo de información diaria: calificaciones de simulacros, horas de estudio, temas dominados y material de repaso, entre otros. Sin un sistema estructurado, esta información se pierde o se maneja de forma desordenada, lo que impide que el estudiante conozca su rendimiento real. El proyecto PrepaTrack soluciona este problema mediante el diseño e implementación de una base de datos relacional que transforma registros aislados en reportes estadísticos exactos.
El presente informe final consolida el trabajo desarrollado a lo largo del proyecto, integrando el primer avance (análisis, esquema conceptual y primer diagrama Entidad-Relación) con el segundo avance (Modelo Entidad-Relación Extendido, bosquejo del modelo relacional y normalización). A partir del diagrama Entidad-Relación inicial, el documento detalla la evolución hacia un Modelo Entidad-Relación Extendido (MEER) y el proceso de normalización, que justifica teóricamente cómo se organizaron las tablas para evitar la duplicidad de información y asegurar el rendimiento del sistema.
Finalmente, se presenta el modelo relacional definitivo, las consultas de aplicación propuestas y el catálogo de datos, dejando la base de datos lista para su implementación física. De este modo, el documento recorre el análisis para el modelo de datos, el esquema conceptual, las reglas de negocio, el diseño lógico y la normalización hasta la Tercera Forma Normal (3FN).
2. Justificación
El desarrollo de PrepaTrack requiere una base de datos relacional porque la información del estudiante está interconectada: un alumno lleva varios cursos, que se dividen en temas y se evalúan mediante simulacros. Para manejar esta complejidad sin inconsistencias, se aplican las reglas de normalización, lo que permite organizar los datos evitando repeticiones innecesarias y asegurando que, si se modifica un registro, el cambio se refleje en todo el sistema sin perder la conexión con los exámenes o materiales de estudio.
Otro motivo importante para usar este modelo es el rendimiento del sistema a futuro. Con el paso de los meses, el volumen de información crecerá rápidamente al registrarse nuevas sesiones de estudio, notas y tarjetas de repaso (flashcards). Al contar con un esquema relacional bien diseñado, con sus llaves primarias y foráneas correctamente definidas, se garantiza que la base de datos soporte este aumento de información sin degradar el tiempo de respuesta de búsquedas y consultas.
Por último, este diseño permite aprovechar las funciones propias del gestor de base de datos para automatizar el trabajo. En lugar de procesar todo de forma externa, se emplearán herramientas nativas de SQL, como las vistas (views) para armar reportes de rendimiento de manera rápida y los disparadores (triggers) para actualizar automáticamente el porcentaje de avance del estudiante cada vez que termine de estudiar un tema. Así, la base de datos no solo almacena la información, sino que ejecuta los cálculos de forma eficiente.
3. Objetivos
3.1. Objetivo General
Diseñar y estructurar una base de datos relacional optimizada para el sistema PrepaTrack, garantizando que el registro del progreso y del material académico sea exacto, consistente y sin redundancias.
3.2. Objetivos Específicos
Definir las tablas principales del sistema, estableciendo correctamente las relaciones y cardinalidades entre ellas.
Normalizar todas las tablas del esquema relacional para evitar la duplicidad de datos y optimizar el almacenamiento de la información.
Estructurar el modelo para facilitar el uso de consultas SQL complejas y vistas (views) que permitan agrupar datos y generar reportes de calificaciones de forma eficiente.
Aplicar mecanismos de integridad referencial y restricciones de dominio que aseguren la validez de los datos almacenados.
4. Análisis para el Modelo de Datos
El análisis para el modelo de datos parte de la identificación de los requerimientos del sistema, los cuales orientan la definición de entidades, atributos y relaciones. A continuación, se presentan los requerimientos funcionales y no funcionales que delimitan el alcance de PrepaTrack. Las reglas de negocio que se derivan de este análisis se detallan en la Sección 6.
4.1. Requerimientos Funcionales
Gestión de cursos y temas
El sistema debe permitir al estudiante registrar cursos y sus respectivos temas.
Cada tema debe poder asignarse a un estado de avance: No iniciado, En proceso, Estudiado, Reforzar o Dominado.
El sistema debe calcular automáticamente el porcentaje de avance por curso.
El sistema debe identificar los temas pendientes y los temas a reforzar.
Simulacros
El sistema debe permitir registrar simulacros con fecha, tiempo de duración, puntaje, número de respuestas correctas e incorrectas y archivo adjunto.
El sistema debe mostrar la evolución del estudiante en distintos simulacros (comparación de puntajes y desempeño).
Horarios de estudio
El sistema debe permitir registrar bloques de estudio indicando día, hora, curso y tema.
Documentos académicos
El sistema debe permitir almacenar documentos académicos asociados a los temas, tales como temarios, separatas, resúmenes y guías.
El sistema debe permitir consultar y organizar estos documentos por curso y tema.
Flashcards
El sistema debe permitir crear y gestionar flashcards por tema.
Cada flashcard debe contener una pregunta y su respectiva respuesta.
Metas de estudio
El sistema debe permitir al estudiante definir metas de estudio (por ejemplo, terminar un conjunto de temas o mejorar puntajes en simulacros).
El sistema debe registrar el estado de cumplimiento de cada meta.
Sesiones de estudio
El sistema debe registrar sesiones de estudio con horas, minutos y segundos invertidos.
Cada sesión debe asociarse a un tema (y, a través de este, a un curso).
El sistema debe registrar el nivel de comprensión alcanzado en cada sesión.
4.2. Requerimientos No Funcionales
El sistema debe calcular porcentajes de avance y estadísticas de simulacros en tiempo real.
Las consultas de documentos, flashcards y sesiones deben responder en menos de tres segundos.
La interfaz debe ser intuitiva y permitir al estudiante registrar cursos y temas sin necesidad de capacitación extensa.
Los registros no deben perderse aunque el usuario cierre sesión inesperadamente.
El sistema debe ser accesible desde la vía web.
El sistema debe autenticar a los usuarios mediante credenciales seguras.
5. Esquema Conceptual
El esquema conceptual define las entidades del sistema, las relaciones entre ellas y los metadatos asociados, sirviendo de base para la elaboración del diagrama Entidad-Relación.
5.1. Entidades
El sistema se compone de diez entidades: Usuario (Estudiante), Curso, Tema, Progreso, Simulacro, Horario, Documento, Flashcard, Meta y Sesión.
5.2. Relaciones
Usuario lleva Curso (1:N): un usuario lleva varios cursos.
Curso contiene Tema (1:N): un curso contiene varios temas.
Tema se refuerza con Flashcard (1:N): un tema tiene varias flashcards.
Tema posee Documento (1:N): un tema posee varios documentos.
Usuario resuelve Simulacro (1:N): un usuario resuelve varios simulacros.
Usuario organiza Horario (1:N): un usuario organiza varios horarios.
Usuario define Meta (1:N): un usuario define varias metas.
Usuario realiza Sesión (1:N): un usuario realiza varias sesiones de estudio.
Usuario lleva Progreso (1:1): un usuario tiene un único registro de progreso.

5.4. Restricciones
El correo del usuario debe ser único en el sistema.
El atributo estado de la entidad Tema solo acepta cinco valores: No iniciado, En proceso, Estudiado, Reforzar y Dominado.
El atributo estado de la entidad Meta solo acepta tres valores: Pendiente, En proceso y Cumplida.
El atributo porcentaje_avance de Progreso debe estar en el rango de 0 a 100.
El atributo nivel_comprensión de Sesión debe estar en el rango de 1 a 5.
Cada Tema debe pertenecer a un único Curso (no puede compartirse entre cursos).

6. Reglas de Negocio
Las reglas de negocio describen las condiciones y comportamientos que el sistema debe respetar de acuerdo con la lógica del dominio de la preparación preuniversitaria.
El estudiante registra cursos y temas, y cada tema tiene diferentes estados de acuerdo con el avance (No iniciado, En proceso, Estudiado, Reforzar, Dominado).
El sistema permite calcular porcentajes de avance por curso, temas pendientes y temas a reforzar.
El usuario puede dar simulacros que se registran con fecha, tiempo, puntaje, preguntas correctas e incorrectas, y guarda el archivo adjunto. Además, se permite ver la evolución del usuario en distintos simulacros.
Se pueden registrar bloques de estudio con día, hora, curso y tema.
El sistema almacena documentos académicos como temarios, separatas, resúmenes y guías.
El sistema cuenta con flashcards o tarjetas de estudio por cada tema, donde cada una tiene su respectiva pregunta y respuesta.
El estudiante puede crear sus metas de estudio, como terminar temas o mejorar puntajes.
El estudiante registra sesiones de estudio, donde se almacenan las horas, minutos y segundos invertidos, además del tema y el nivel de comprensión alcanzado.

Respecto al primer diagrama, se realizaron ajustes a partir de la retroalimentación recibida en el informe anterior, mejorando la organización de los recursos académicos y la definición de las cardinalidades.
En esta etapa se consolidan las relaciones jerárquicas y de dependencia entre las entidades: el Usuario concentra las relaciones con Curso, Simulacro, Horario, Meta, Sesión y Progreso, mientras que el Tema articula los recursos de estudio (Documento y Flashcard) y se vincula con el Curso al que pertenece.

9.4. Tercera Forma Normal (3FN)
Se observa la tabla Documento resultante de la 2FN, que contiene tanto id_tema como id_curso. Esto genera una dependencia transitiva: por regla de negocio, un documento pertenece a un tema y ese tema ya pertenece a un curso. Si se conoce el id_tema, indirectamente se conoce el id_curso. Almacenar id_curso en la tabla Documento es redundante en 3FN, por lo que se elimina. Las demás tablas creadas en la 2FN mantienen su forma, dado que no presentan dependencias transitivas.

10. Consultas de Aplicación
La base de datos del sistema PrepaTrack se implementará en MySQL, utilizando el motor de almacenamiento InnoDB para garantizar la integridad referencial mediante llaves foráneas. A continuación se presentan diecisiete consultas de aplicación, redactadas en sintaxis MySQL y alineadas con los requerimientos funcionales y las reglas de negocio del sistema. Constituyen una propuesta sujeta a validación durante la implementación física. Al final se incluyen los objetos de base de datos (una vista y un disparador) que automatizan el cálculo del avance y la generación de reportes. En las consultas se emplea como ejemplo el identificador de usuario 1 (id_usuario = 1).
10.1. Porcentaje de avance por curso
Calcula el porcentaje de temas dominados respecto al total de temas de cada curso de un estudiante. Se aprovecha que MySQL evalúa una condición booleana como 1 o 0.
SELECT  c.nombre AS curso,
        COUNT(t.id_tema) AS total_temas,
        SUM(t.estado = 'Dominado') AS temas_dominados,
        ROUND(SUM(t.estado = 'Dominado') * 100 / COUNT(t.id_tema), 2) AS porcentaje_avance
FROM    curso c
JOIN    tema  t ON t.id_curso = c.id_curso
WHERE   c.id_usuario = 1
GROUP BY c.id_curso, c.nombre;
10.2. Temas pendientes o por reforzar
Lista los temas que requieren atención según su estado. La función MySQL FIELD() ordena por prioridad de atención.
SELECT  c.nombre AS curso, t.nombre AS tema, t.estado
FROM    tema  t
JOIN    curso c ON c.id_curso = t.id_curso
WHERE   c.id_usuario = 1
  AND   t.estado IN ('No iniciado', 'En proceso', 'Reforzar')
ORDER BY c.nombre,
         FIELD(t.estado, 'Reforzar', 'En proceso', 'No iniciado');
10.3. Conteo de temas por estado
Agrupa los temas del estudiante según su estado de avance para obtener un resumen rápido.
SELECT  t.estado, COUNT(*) AS cantidad
FROM    tema  t
JOIN    curso c ON c.id_curso = t.id_curso
WHERE   c.id_usuario = 1
GROUP BY t.estado
ORDER BY cantidad DESC;
10.4. Cursos completados al 100%
Identifica los cursos en los que todos los temas se encuentran en estado Dominado, usando HAVING.
SELECT  c.nombre AS curso
FROM    curso c
JOIN    tema  t ON t.id_curso = c.id_curso
WHERE   c.id_usuario = 1
GROUP BY c.id_curso, c.nombre
HAVING  SUM(t.estado <> 'Dominado') = 0;
10.5. Evolución de puntajes en simulacros
Muestra la progresión del desempeño de un estudiante a lo largo del tiempo.
SELECT  fecha, puntaje, tiempo
FROM    simulacro
WHERE   id_usuario = 1
ORDER BY fecha ASC;
10.6. Estadísticas de simulacros por usuario
Calcula el promedio, el mejor y el peor puntaje de cada usuario. El LEFT JOIN incluye usuarios sin simulacros.
SELECT  u.nombre,
        COUNT(s.id_simulacro) AS total_simulacros,
        ROUND(AVG(s.puntaje), 2) AS promedio,
        MAX(s.puntaje) AS mejor_puntaje,
        MIN(s.puntaje) AS peor_puntaje
FROM    usuario u
LEFT JOIN simulacro s ON s.id_usuario = u.id_usuario
GROUP BY u.id_usuario, u.nombre;
10.7. Simulacros de los últimos 30 días
Filtra los simulacros recientes empleando funciones de fecha de MySQL.
SELECT  fecha, puntaje
FROM    simulacro
WHERE   id_usuario = 1
  AND   fecha >= CURDATE() - INTERVAL 30 DAY
ORDER BY fecha DESC;
10.8. Mejor simulacro de cada usuario
Obtiene, para cada usuario, el simulacro con mayor puntaje mediante una subconsulta correlacionada.
SELECT  s.id_usuario, s.fecha, s.puntaje
FROM    simulacro s
WHERE   s.puntaje = (
          SELECT MAX(s2.puntaje)
          FROM   simulacro s2
          WHERE  s2.id_usuario = s.id_usuario);
10.9. Documentos disponibles por curso y tema
Une documento, tema y curso para organizar el material académico del estudiante.
SELECT  c.nombre AS curso, t.nombre AS tema,
        d.temario, d.separata, d.resumen, d.guia
FROM    documento d
JOIN    tema  t ON t.id_tema  = d.id_tema
JOIN    curso c ON c.id_curso = t.id_curso
WHERE   c.id_usuario = 1
ORDER BY c.nombre, t.nombre;
10.10. Cantidad de flashcards por tema
Cuenta las flashcards asociadas a cada tema. El LEFT JOIN muestra también los temas sin flashcards.
SELECT  t.nombre AS tema, COUNT(f.id_flashcard) AS total_flashcards
FROM    tema t
LEFT JOIN flashcard f ON f.id_tema = t.id_tema
GROUP BY t.id_tema, t.nombre
ORDER BY total_flashcards DESC;
10.11. Flashcards de un tema específico
Recupera las preguntas y respuestas de un tema para una sesión de repaso.
SELECT  pregunta, respuesta
FROM    flashcard
WHERE   id_tema = 1;
10.12. Tiempo total de estudio por curso
Suma las horas, minutos y segundos de las sesiones y los convierte a formato de tiempo con SEC_TO_TIME() de MySQL.
SELECT  c.nombre AS curso,
        SEC_TO_TIME(
          SUM(se.tiempo_hora * 3600 + se.tiempo_minuto * 60 + se.tiempo_segundo)
        ) AS tiempo_total
FROM    sesion se
JOIN    tema  t ON t.id_tema  = se.id_tema
JOIN    curso c ON c.id_curso = t.id_curso
WHERE   se.id_usuario = 1
GROUP BY c.id_curso, c.nombre;
10.13. Nivel de comprensión promedio por tema
Promedia el nivel de comprensión registrado en las sesiones de estudio de cada tema.
SELECT  t.nombre AS tema,
        ROUND(AVG(se.nivel_comprension), 2) AS comprension_promedio
FROM    sesion se
JOIN    tema  t ON t.id_tema = se.id_tema
GROUP BY t.id_tema, t.nombre
ORDER BY comprension_promedio ASC;
10.14. Temas que requieren refuerzo por baja comprensión
Filtra los temas cuyo nivel de comprensión promedio es menor a 3, usando HAVING.
SELECT  t.nombre AS tema,
        ROUND(AVG(se.nivel_comprension), 2) AS comprension_promedio
FROM    sesion se
JOIN    tema  t ON t.id_tema = se.id_tema
GROUP BY t.id_tema, t.nombre
HAVING  AVG(se.nivel_comprension) < 3
ORDER BY comprension_promedio ASC;
10.15. Estado de cumplimiento de metas por usuario
Agrupa las metas por estado para conocer cuántas están pendientes, en proceso o cumplidas.
SELECT  u.nombre, m.estado, COUNT(*) AS cantidad
FROM    meta m
JOIN    usuario u ON u.id_usuario = m.id_usuario
GROUP BY u.id_usuario, u.nombre, m.estado
ORDER BY u.nombre, m.estado;
10.16. Horario de estudio semanal de un usuario
Ordena los bloques de estudio por día de la semana y hora mediante FIELD() de MySQL.
SELECT  dia, hora
FROM    horario
WHERE   id_usuario = 1
ORDER BY FIELD(dia, 'Lunes', 'Martes', 'Miércoles',
               'Jueves', 'Viernes', 'Sábado', 'Domingo'),
         hora;
10.17. Progreso global de todos los usuarios
Muestra el porcentaje de avance global de cada estudiante, ordenado de mayor a menor.
SELECT  u.nombre, p.porcentaje_av AS porcentaje_avance
FROM    usuario u
JOIN    progreso p ON p.id_usuario = u.id_usuario
ORDER BY p.porcentaje_av DESC;
10.18. Objetos de base de datos (MySQL)
Además de las consultas anteriores, el sistema define una vista y un disparador en MySQL para automatizar tareas frecuentes.
Vista de reporte de rendimiento
Vista (view) que consolida el rendimiento por usuario y curso para la generación rápida de reportes.
CREATE OR REPLACE VIEW vista_rendimiento AS
SELECT  u.id_usuario, u.nombre, c.nombre AS curso,
        COUNT(t.id_tema) AS total_temas,
        SUM(t.estado = 'Dominado') AS dominados
FROM    usuario u
JOIN    curso  c ON c.id_usuario = u.id_usuario
JOIN    tema   t ON t.id_curso  = c.id_curso
GROUP BY u.id_usuario, u.nombre, c.id_curso, c.nombre;
Disparador para actualizar el progreso
Disparador (trigger) que recalcula el porcentaje de avance del estudiante cada vez que cambia el estado de un tema.
DELIMITER //
CREATE TRIGGER trg_actualiza_progreso
AFTER UPDATE ON tema
FOR EACH ROW
BEGIN
  DECLARE v_usuario INT;
  SELECT id_usuario INTO v_usuario
  FROM   curso WHERE id_curso = NEW.id_curso;
  UPDATE progreso
  SET    porcentaje_av = (
           SELECT ROUND(SUM(t.estado = 'Dominado') * 100 / COUNT(t.id_tema), 2)
           FROM   tema  t
           JOIN   curso c ON c.id_curso = t.id_curso
           WHERE  c.id_usuario = v_usuario)
  WHERE  id_usuario = v_usuario;
END //
DELIMITER ;
11. Conclusiones
El diseño relacional propuesto para PrepaTrack cumple con el objetivo general del proyecto, al organizar la información académica del estudiante en diez entidades normalizadas que garantizan exactitud, consistencia y ausencia de redundancias.
La evolución desde el diagrama Entidad-Relación inicial hacia el Modelo Entidad-Relación Extendido permitió consolidar las cardinalidades y las relaciones jerárquicas, situando al Usuario como entidad central y al Tema como articulador de los recursos de estudio.
El proceso de normalización hasta la Tercera Forma Normal eliminó los grupos repetidos (1FN), las dependencias parciales (2FN) y la dependencia transitiva detectada en la tabla Documento (3FN), mejorando la integridad y el rendimiento del modelo.
El catálogo de datos y el modelo relacional definitivo dejan la base de datos lista para su implementación física en MySQL (motor InnoDB), mientras que las vistas y los disparadores propuestos permitirán automatizar el cálculo del avance y la generación de reportes.
Como trabajo pendiente, se debe insertar la versión en escala de grises de los diagramas (Figuras 1, 2 y 3) e implementar y validar las diecisiete consultas de aplicación, la vista y el disparador sobre el gestor MySQL.
